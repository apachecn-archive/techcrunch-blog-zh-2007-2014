# Twitter 为第三方开发者推出了问题跟踪器和其他新功能

九月初，Twitter 的杰克·多西[向开发者发布了一份通知](https://web.archive.org/web/20230204224558/https://techcrunch.com/2011/09/01/twitters-dorsey-to-developers-with-apple-we-can-take-this-to-a-whole-new-level/),就网络如何更好地为他们服务征求反馈。来自帖子:*我们想知道你需要我们提供哪些额外的材料来帮助你生产产品、促进销售和扩大你的影响范围。*看起来 Twitter 已经考虑到了这些反馈，并且[已经根据开发者的反馈为他们推出了](https://web.archive.org/web/20230204224558/https://dev.twitter.com/blog/acting-your-feedback)许多新功能。

开发人员的主要需求之一是集中式问题跟踪器。Twitter 现在推出了一个专门的问题跟踪器，托管在其新的开发者门户网站上，它将取代现有的谷歌代码跟踪器。

另一个反馈是关于 API 变更的明确政策。Twitter 现在将在任何东西“失效”之前至少提前 30 天通知开发者。对于更广泛、更彻底的改变，Twitter 表示将灵活对待开发者，给他们尽可能多的时间。

速率限制[对于 Twitter 开发者来说是一个大问题](https://web.archive.org/web/20230204224558/https://techcrunch.com/2010/08/30/twitter-site-streams-beta/)，该公司表示，如果你正在进行认证呼叫，它允许每个用户每小时 350 个请求(oauth_token ),对于未经认证的呼叫，每小时 150 个请求。

我们的目标是随着开发者用户群的增长来扩展 API 的使用，随着他们带来更多的用户，他们将获得更多访问 APU 的机会。Twitter 还表示，它致力于推动网站流，这将允许应用程序接收事件的实时更新，如提及、关注、时间线等，这是 2011 年 Q1 推出的测试版。

Twitter 承诺与开发者进行更多的沟通，尤其是帮助第三方开发者解决使用 oAuth 的认证问题。

通过投资为开发者创造新的工具和功能，Twitter 显然正在与第三方开发者进行更加协调一致的合作。在过去，已经出现了沟通不畅的情况，网络开始倾听开发者想要什么和需要什么是明智的。三月份，Twitter [基本上告诉开发者](https://web.archive.org/web/20230204224558/https://techcrunch.com/2011/03/11/twitter-ecosystem-guidelines/)避免在原生客户端上与他们竞争。这并不是说 Twitter 不想让开发者构建他们的平台，他们只是不想让开发者构建模仿 Twitter 自己服务的客户端。正如我们所写的，这是[的生意，不是个人的。](https://web.archive.org/web/20230204224558/https://techcrunch.com/2011/02/19/this-is-business-not-personal/)不管怎样，很高兴看到 Twitter 采取行动与开发者和解。
